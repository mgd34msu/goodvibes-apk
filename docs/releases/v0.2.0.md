# GoodVibes Companion v0.2.0

Separate per-chat model selection and brand-level provider routing.

## Highlights

- Companion remote chat sessions now carry their own provider/model — switching the model for a remote chat does not change the TUI/global model, and vice versa
- New "Apply Selection To" picker (Global / This Chat) on the Models tab makes the scope of every model switch explicit
- Provider list now exposes auth-route metadata (api-key, subscription-oauth, etc.) with freshness/usability — OpenAI subscription auth is recognized for companion-facing clients
- The companion app no longer needs to know about provider sub-variants (e.g. `openai-codex`); selecting `openai` lets the daemon route between API key and subscription internally
- Bumps `@pellux/goodvibes-sdk` to **0.25.8**
- Adds the missing `@react-native/typescript-config` dev dependency so `npm run typecheck` actually runs against the project

## Behavior Notes

- Shared/TUI session model changes still go through `PATCH /api/providers/current` (intentionally global)
- Remote chat model changes go through `PATCH /api/companion/chat/sessions/{sessionId}` (session-local)
- New chats created from the companion stamp the staged "This Chat" selection at creation time; if nothing is staged the daemon picks its default
- The previous "this chat is pinned to a different model than the global picker" warning has been removed — divergence between the two scopes is now the expected behavior

## Setup

Unchanged from v0.1.0:

- a running GoodVibes daemon (with the matching SDK release)
- a reachable daemon URL such as `http://192.168.x.x:3210`
- daemon username/password credentials or a bearer token with the scopes you need

## Known Constraints

- No iOS build artifact in this release
- Phone must be able to reach the daemon URL directly
- Approval, session-write, and chat behavior still depend on the scopes granted by the daemon auth in use
